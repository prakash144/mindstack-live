    🧠 JVM Memory Model¶

    The Java Virtual Machine (JVM) memory model is divided into several memory areas, each responsible for different aspects of program execution and memory management. Understanding these areas is critical for performance tuning, debugging memory leaks, and writing efficient, scalable Java applications.

    📦 1. Heap Memory¶

    • Purpose: Stores all objects and class instances.
    • Lifecycle: Exists as long as the JVM process is alive.
    • Managed by: Garbage Collector (GC) ♻️.
    • Divided into:
      • 🔹 Young Generation: Newly created objects.
        • Eden Space ➕ Survivor Spaces (S0 & S1)
      • 🔹 Old Generation (Tenured): Long-lived objects.
    • Tuning: Critical for reducing GC pauses.
    • GC Algorithms: G1, CMS, ZGC, Shenandoah.
    // Example: Object stored in Heap
Employee emp = new Employee();

    🧮 2. Stack Memory¶

    • Purpose: Stores primitive variables, method calls, and references to objects in heap.
    • Structure: Each thread has its own stack 🧵.
    • Contains:
      • Stack Frames (per method call)
      • Local variables, operand stack, return address
    • Memory is automatically freed after method execution.
    • Faster access than heap memory.
    • Exception: StackOverflowError when recursion is too deep.
    void doSomething() {
    int count = 10; // Stored in Stack
}

    🗃️ 3. Method Area (aka Class Area)¶

    • Purpose: Stores class metadata, static variables, and method code.
    • Shared among all threads.
    • Contents:
      • Runtime constant pool
      • Method and constructor bytecode
      • Field and method data
    • Part of the Metaspace (since Java 8).

    🧩 4. MetaSpace (Java 8+)¶

    • Replaces: PermGen (pre-Java 8).
    • Purpose: Stores class metadata, class definitions, method data.
    • Allocated in native memory (outside heap).
    • Grows dynamically by default (can be tuned).
    • Tuning Flags:
      • -XX:MaxMetaspaceSize
      • -XX:MetaspaceSize

    🧠 Summary Table¶

    Memory Area Stores Thread Scoped GC Managed Allocated In
    🧮 Stack Method calls, local variables ✅ Yes ❌ No JVM Memory
    📦 Heap Objects, class instances ❌ No ✅ Yes JVM Memory
    🗃️ Method Area Class structure, bytecode, statics ❌ No ✅ Yes Part of MetaSpace
    🧩 MetaSpace Class metadata (Java 8+) ❌ No ✅ Yes Native Memory

    🔍 Advanced Considerations¶

    • Use tools like jconsole, jvisualvm, jstat, and MAT to monitor memory.
    • Analyze GC logs for performance bottlenecks.
    • Be cautious of classloader leaks in MetaSpace.
    • JVM flags for fine-tuning can drastically affect latency and throughput.

    📚 Recommended Reads¶

    • Java Performance: The Definitive Guide by Scott Oaks
    • JVM tuning documentation for G1, ZGC, and Shenandoah
    • Oracle's official Java memory tuning guide

    ✅ Mastery of the JVM memory model is essential for backend engineers, especially when scaling microservices or dealing with high-throughput, low-latency systems.
