    SOLID Principles in Java¶

    Overview¶

    This project demonstrates the application of the SOLID principles using Java classes. SOLID is an acronym for five design principles intended to make software designs more understandable, flexible, and maintainable. The principles are:

    1. Single Responsibility Principle (SRP)
    2. Open/Closed Principle (OCP)
    3. Liskov Substitution Principle (LSP)
    4. Interface Segregation Principle (ISP)
    5. Dependency Inversion Principle (DIP)

    Single Responsibility Principle (SRP)¶

                                  +-------------------------+
                              |   SingleResponsibility  |
                              |       (Main Class)      |
                              +-------------------------+
                                           |
                  +------------------------+------------------------+
                  |                                                 |
    +------------------------+                      +-------------------------+
    |        Marker          |                      |        Invoice          |
    |   (Holds marker data)  |                      | (Calculates total cost) |
    +------------------------+                      +-------------------------+
    |                                                           |
    +------------------------+                      +-------------------------+
    | - getName()            |                      | - getMarker()           |
    | - getColor()           |                      | - getQuantity()         |
    | - getYear()            |                      | - calculateTotal()      |
    | - getPrice()           |                      +-------------------------+
    +------------------------+                                
    +-------------------------+
    |      InvoiceDao         |
    |  (Saves to Database)    |
    +-------------------------+
    | - saveToDatabase()      |
    +-------------------------+
    |
    +-------------------------+
    |    InvoicePrinter       |
    |  (Prints Invoice Info)  |
    +-------------------------+
    | - print()               |
    +-------------------------+

    Class Descriptions¶

    • Marker: Represents a marker with properties like name, color, year, and price. It has a single responsibility of managing marker information.
    • Invoice: Manages the relationship between a Marker and its quantity. It calculates the total price based on the marker's price and quantity.
    • InvoiceDao: Handles the persistence of the Invoice data, such as saving it to a database.
    • InvoicePrinter: Prints invoice details to the console, adhering to the SRP by handling only printing logic.

    Open/Closed Principle (OCP)¶

                                +---------------------+
                            |    InvoiceDao       |
                            |  (Interface)        |
                            +---------------------+
                                      |
      +-------------------------------+------------------------------+
      |                                                              |
+------------------------+                               +------------------------+
|   DatabaseInvoiceDao   |                               |    FileInvoiceDao      |
| (Implements InvoiceDao)|                               | (Implements InvoiceDao)|
+------------------------+                               +------------------------+
            |                                                            |
            v                                                            v
save(Invoice) (Saves to DB)                                   save(Invoice) (Saves to File)

    Classes Involved:¶

    • InvoiceDao (interface)
    • DatabaseInvoiceDao
    • FileInvoiceDao

    The Open/Closed Principle states that software entities should be open for extension but closed for modification. This means that new functionality should be added with new code, without altering existing code.

    Class Descriptions¶

    • InvoiceDao: An interface that declares a method save(Invoice invoice) for saving invoices. Different implementations can extend this interface.
    • DatabaseInvoiceDao: Implements InvoiceDao to save invoices to a database.
    • FileInvoiceDao: Implements InvoiceDao to save invoices to a file.
    • Main Class: OpenClosed_2 demonstrates the use of different InvoiceDao implementations without changing the class itself.

    Liskov Substitution Principle (LSP)¶

                          +------------------------+
                      |     easy.educativeIO.Vehicle            |
                      |  + getNumberOfWheels() |
                      +------------------------+
                             /     \
       +----------------+   +------------------+
       | EngineVehicle  |   |     Bicycle      |
       | + hasEngine()  |   |  (No Engine)     |
       | (4 wheels)     |   |  (2 wheels)      |
       +----------------+   +------------------+
         /        \
+---------------+ +---------------+
|      easy.educativeIO.Car      | |  MotorCycle   |
| (inherits EV) | | (inherits EV) |
+---------------+ +---------------+

    Classes Involved:¶

    • easy.educativeIO.Vehicle (Base class)
    • EngineVehicle (Derived class)
    • easy.educativeIO.Car, MotorCycle, Bicycle (Subclasses)

    The Liskov Substitution Principle states that objects of a derived class should be able to replace objects of the base class without affecting the correctness of the program.

    Class Descriptions¶

    • easy.educativeIO.Vehicle: A base class representing general vehicles with a method getNumberOfWheels.
    • EngineVehicle: Extends easy.educativeIO.Vehicle to include engine-specific behavior. Represents vehicles with engines.
    • easy.educativeIO.Car and MotorCycle: Extend EngineVehicle, inheriting properties of vehicles with engines.
    • Bicycle: Extends easy.educativeIO.Vehicle without adding or modifying behavior related to engines, adhering to LSP.

    Interface Segregation Principle (ISP)¶

    +--------------------+                +--------------------+
| WaiterInterface    |                | ChefInterface      |
+--------------------+                +--------------------+
| + serve()          |                | + cookFood()       |
| + takeOrder()      |                | + decideMenu()     |
+--------------------+                +--------------------+
           |                                    |
+------------------+                  +------------------+
| Waiter           |                  | Chef             |
+------------------+                  +------------------+
| + serve()        |                  | + cookFood()     |
| + takeOrder()    |                  | + decideMenu()   |
+------------------+                  +------------------+

    This principle ensures that no client is forced to depend on methods it does not use.

    Explanation¶

    The WaiterInterface and ChefInterface segregate responsibilities, ensuring that the Waiter and Chef classes implement only the methods they need.

    Has-a Relationship:¶

    • Waiter has actions like serve and takeOrder.
    • Chef has actions like cookFood and decideMenu.

    Is-a Relationship:¶

    • Waiter is a WaiterInterface.
    • Chef is a ChefInterface.

    Dependency Inversion Principle (DIP)¶

    +--------------------+         +--------------------+
|   Keyboard         |         |      Mouse         |
+--------------------+         +--------------------+
| + type()           |         | + click()          |
+--------------------+         +--------------------+
/      \                      /       \
+---------------+         +------------------------+
| WiredKeyboard |         |      WiredMouse        |
+---------------+         +------------------------+
| + type()      |         | + click()              |
+---------------+         +------------------------+

+-----------------------+    +---------------------------+
|  BluetoothKeyboard    |    |     BluetoothMouse        |
+-----------------------+    +---------------------------+
| + type()              |    | + click()                 |
+-----------------------+    +---------------------------+

+-------------------------------------------------------+
|                        MacBook                        |
+-------------------------------------------------------+
| - keyboard: Keyboard                                  |
| - mouse: Mouse                                        |
+-------------------------------------------------------+
| + useLaptop()                                         |
+-------------------------------------------------------+

    This principle states that high-level modules should depend on abstractions rather than concrete implementations.

    Explanation¶

    The MacBook class depends on the abstractions Keyboard and Mouse rather than specific implementations, allowing flexible configuration with different peripheral types.

    Has-a Relationship:¶

    • MacBook has a Keyboard and a Mouse.

    Is-a Relationship:¶

    • WiredKeyboard and BluetoothKeyboard are types of Keyboard.
    • WiredMouse and BluetoothMouse are types of Mouse.
