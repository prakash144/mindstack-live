    🔹 System Calls Overview¶

    System Calls serve as an interface between user applications and the operating system kernel, enabling secure access to hardware and resources.

    🔹 Real-Life Analogy¶

    📞 System Call as a Customer Service Representative: - A user program (customer) requests a service (e.g., open a file, allocate memory). - The OS (company) processes the request via a system call (service rep). - The service rep returns the result safely (grants or denies access).

    🔹 How System Calls Work¶

    🛠 User Mode → Kernel Mode Transition: 1. User Mode Execution: Applications operate in a restricted environment. 2. System Call Invocation: A request is made (e.g., open() file). 3. Mode Switch: Control shifts to kernel mode. 4. Request Processing: The OS validates, executes, and manages resources. 5. Result Delivery: Control returns to user mode with a response.

    🔹 Categories of System Calls¶

    Category Description & Examples
    Process Control ⚙️ Create, execute, terminate processes.
    🔹 fork(), exec(), wait()
    File Management 📂 Open, read, write, close files.
    🔹 open(), read(), write(), close()
    Device Management 🖥️ Interact with hardware devices.
    🔹 ioctl(), read(), write()
    Information Maintenance 📊 Retrieve system/process details.
    🔹 getpid(), gettimeofday()
    Communication (IPC) 🔄 Enable process communication.
    🔹 pipe(), socket(), send(), recv()

    🔹 Importance of System Calls¶

    ✅ Security & Stability: Restrict direct hardware access. ✅ Hardware Abstraction: Provide a uniform interface across devices. ✅ Controlled Resource Management: Ensure fair access to CPU, memory, devices.

    🔹 Considerations & Limitations¶

    ⚠️ Overhead: Frequent system calls can slow performance. ⚠️ Complexity: Requires OS knowledge to use effectively. ⚠️ Portability Issues: OS-specific system calls affect cross-platform applications.

    🔹 Example Scenario¶

    📄 Text Editor Saving a File: 1. open(): Opens the file. 2. write(): Writes user data. 3. close(): Finalizes changes.

    🚀 System Boot Process¶

    The system boot process is a sequence of steps that initialize the hardware and software to bring the system to an operational state.

    🔹 Real-Life Analogy¶

    🏢 Restaurant Opening for the Day: - 🔍 Check Equipment (POST) 🛠️ - 📋 Load Initial Setup (BIOS/UEFI) 🖥️ - 🔄 Start Main Operations (Kernel) 🚀 - 🔌 Prepare Services (Init/Systemd) 🏗️

    🔹 Key Stages¶

    Stage Description
    1️⃣ POST (Power-On Self-Test) Firmware checks hardware components (RAM, CPU, disks).
    2️⃣ BIOS/UEFI Initialization Identifies bootable devices & prepares to load OS.
    3️⃣ Bootloader Execution Loads the OS kernel into memory (e.g., GRUB, Windows Boot Manager).
    4️⃣ Kernel Initialization Manages hardware, sets up memory structures, and loads drivers.
    5️⃣ Init/System Services Startup Launches essential system processes (networking, logging, user session management).

    🔹 Importance of Understanding the Boot Process¶

    ✅ Troubleshooting: Identify startup failures. ✅ Performance Optimization: Disable unnecessary startup services. ✅ Security: Secure Boot prevents unauthorized OS modifications. ✅ Customization: Modify boot parameters, kernels, and configurations.

    🔹 Legacy BIOS vs. UEFI¶

    Feature BIOS 🏛️ UEFI 🚀
    Memory Addressing 16-bit, limited to 1MB 64-bit, supports large storage
    Boot Device Support MBR (limited to 2TB drives) GPT (supports drives >2TB)
    Security No Secure Boot Secure Boot support
    User Interface Text-based Graphical, mouse support
