    üéØ Purpose: These notes are for quick revision before your Google SSE interview ‚Äî focused on designing secure systems from scratch like a Senior Engineer or Architect would.

    üß† Introduction: How to Think About Security in the Design Phase¶

    Security must be embedded from the very beginning ‚Äî not added later as a patch.

    üß© Start with the Following Principles:¶

    • üîç Threat Modeling:

      • Identify possible entry points and attack surfaces.
      • Tools: STRIDE, DREAD, PASTA models.

    • üé≠ Define Trust Boundaries:

      • Where does data enter and exit the system?
      • Where is it at risk?

    • üõ° Apply Defense-in-Depth:

      • Multiple layers of security (network, app, data, identity).

    • üö´ Least Privilege & Zero Trust:

      • Every service, user, and resource gets only what it needs, nothing more.

    • üß∞ Assume Breach:

      • Design like the system will get breached ‚Äî how will it contain damage?

    • ‚úÖ Security Requirements = Functional Requirements:

      • Define security as part of your acceptance criteria from day one.

    üîê Authentication & Authorization¶

    ‚úÖ Authentication¶

    • Use OAuth 2.0 / OIDC with a centralized Identity Provider (Keycloak, Okta, Auth0).
    • Support flows like:
      • Authorization Code (for web)
      • Client Credentials (for services)

    ‚úÖ Authorization¶

    • Use RBAC or ABAC with clear policies.
    • Use JWTs (short-lived) for identity propagation.
    • Validate token signature, expiration, audience.

    üîí Secure Communication¶

    • Use TLS (HTTPS) everywhere (internal + external).
    • Use mTLS (mutual TLS) for internal service calls.
    • Automate cert renewals (e.g., cert-manager for Kubernetes).
    • Enforce strict protocol versions to avoid downgrade attacks.

    üß± Secret Management¶

    • Use secure secret stores like:
      • HashiCorp Vault, AWS Secrets Manager, Azure Key Vault.
    • üîÅ Rotate secrets and credentials regularly.
    • üìú Audit all access and operations on secrets.

    üì¶ API and Endpoint Security¶

    • Use an API Gateway to:
      • Authenticate requests
      • Enforce rate limits
      • Validate schemas
    • Avoid exposing internal APIs externally.
    • Sanitize and validate all input/output.

    üîÅ Service-to-Service Security¶

    • Use mTLS or signed JWTs for inter-service authentication.
    • Enforce network policies and namespace isolation.
    • Consider a Service Mesh (Istio, Linkerd) to manage security policies.

    üß™ CI/CD Pipeline Security¶

    • Add SAST, DAST, and SCA security scans in CI.
    • Sign build artifacts and verify checksums.
    • Use secure base images.
    • Lock down deployment access.

    üê≥ Container & Runtime Security¶

    • Use minimal base images (e.g., distroless, Alpine).
    • Run containers as non-root.
    • Apply Pod Security Policies or PSA (Kubernetes).
    • Enable runtime detection tools like Falco.

    üîê Data Security¶

    • üîí Encrypt sensitive data:
      • At Rest (AES-256)
      • In Transit (TLS)
    • Use managed KMS for key rotation.
    • Mask PII in logs and test environments.
    • Adhere to compliance standards (GDPR, HIPAA, etc.).

    üìä Monitoring, Logging & Response¶

    • Centralized logging with tools like ELK, Loki, Fluentbit.
    • Integrate SIEM tools (e.g., Splunk, Datadog).
    • Configure alerts for anomalies or brute force.
    • Maintain an Incident Response Plan.

    üî• Must-Know Concepts for Interviews¶

    üîÅ Zero Trust Architecture¶

    • Never trust; always verify.
    • Fine-grained access, continuous validation.

    üßæ OAuth vs OIDC¶

    • OAuth ‚Üí Delegated authorization
    • OIDC ‚Üí Adds authentication layer

    üîê JWT Best Practices¶

    • Use short expiration + refresh.
    • Never store in localStorage (use HttpOnly cookies).
    • Validate iss, aud, exp.

    üß± Defense in Depth¶

    • Layered security across network, host, app, data.

    üßº OWASP Top 10¶

    • SQLi, XSS, CSRF, SSRF, Broken Auth, etc.

    ‚úÖ Final Checklist for Security-First Design¶

    Area Must-Have Security Practices
    Identity OAuth2 + OIDC, Centralized IdP
    Transport TLS + mTLS, Enforced Protocols
    Secrets Secret Manager, Rotation, No Hardcoding
    APIs Gateway Auth, Input Sanitization, Rate Limits
    Services mTLS, Network Policies, Scoped JWTs
    Containers Non-root, Minimal Base, Runtime Protection
    Data AES-256 + KMS, Masking, Compliance
    CI/CD SAST/DAST, Signed Builds, Access Control
    Monitoring Central Logs, Alerts, SIEM Integration
