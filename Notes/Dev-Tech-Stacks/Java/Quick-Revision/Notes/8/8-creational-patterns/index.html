    8 creational patterns

    üîß Creational Design Patterns (GoF)¶

    üìå Purpose:¶

    Deal with object creation mechanisms, aiming to increase flexibility and reuse of existing code.

    üß± 1. Singleton Pattern üîí¶

    • Ensures only one instance of a class exists.
    • Provides a global access point to it.

    ‚úÖ Use Case:¶

    • Logging, configuration management, DB connections.

    üß† Key Concepts:¶

    • Private constructor
    • Static instance reference
    • Thread-safe lazy initialization

    üí° Example (Java):¶

    public class Singleton {
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}

    üè≠ 2. Factory Pattern üèóÔ∏è¶

    • Delegates object creation to subclasses or factory classes.
    • Hides the logic of instantiation from the client.

    ‚úÖ Use Case:¶

    • When object creation involves complex logic or conditions.

    üß† Key Concepts:¶

    • Abstract class/interface for product
    • Factory method to decide which object to instantiate

    üí° Example:¶

    interface Shape {
    void draw();
}
class Circle implements Shape { public void draw() { System.out.println("Circle"); } }
class Square implements Shape { public void draw() { System.out.println("Square"); } }

class ShapeFactory {
    public Shape getShape(String type) {
        return switch (type) {
            case "circle" -> new Circle();
            case "square" -> new Square();
            default -> throw new IllegalArgumentException("Unknown shape");
        };
    }
}

    üß∞ 3. Builder Pattern üèóÔ∏èüì¶¶

    • Separates construction of a complex object from its representation.
    • Allows step-by-step object creation.

    ‚úÖ Use Case:¶

    • Creating objects with many optional parameters (e.g., StringBuilder, HTTP requests).

    üß† Key Concepts:¶

    • Immutable objects
    • Avoiding constructor telescoping
    • Fluent interface design

    üí° Example:¶

    class User {
    private final String name;
    private final int age;
    private final String address;

    private User(Builder builder) {
        this.name = builder.name;
        this.age = builder.age;
        this.address = builder.address;
    }

    static class Builder {
        private String name;
        private int age;
        private String address;

        Builder setName(String name) { this.name = name; return this; }
        Builder setAge(int age) { this.age = age; return this; }
        Builder setAddress(String address) { this.address = address; return this; }
        User build() { return new User(this); }
    }
}

    üß¨ 4. Prototype Pattern üß™¶

    • Creates new objects by copying existing ones (cloning).
    • Avoids costly operations like new for complex objects.

    ‚úÖ Use Case:¶

    • When object creation is resource-intensive (e.g., parsing config, DB load).

    üß† Key Concepts:¶

    • Implements Cloneable and overrides clone()
    • Deep vs shallow copy

    üí° Example:¶

    class Document implements Cloneable {
    String content;

    public Document(String content) {
        this.content = content;
    }

    public Document clone() {
        return new Document(this.content);
    }
}

    üìö Comparison Table¶

    Pattern Intent Key Feature Thread-Safety/Complexity
    Singleton One instance only Global point of access Needs synchronization
    Factory Create object based on input Decouples creation logic Moderate
    Builder Construct complex object step-by-step Immutable object creation Low
    Prototype Clone an object Avoids costly instantiation Depends on clone logic

    üéØ Pro Tips (For Senior Devs)¶

    • üßµ Prefer enum Singleton in Java for serialization and thread safety.
    • üß© Factory is good for polymorphic behavior, not just object creation.
    • üß± Combine Builder + Factory for highly configurable products.
    • üß™ Avoid using Prototype when deep copy becomes too error-prone.

    ‚úÖ These patterns are fundamental to writing flexible, maintainable, and testable software. Use them wisely based on the context and constraints of your project.
