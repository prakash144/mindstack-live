    🤖 Behavioral Design Patterns¶

    📌 Overview¶

    • Behavioral patterns define how objects interact and communicate with each other.
    • Focus on responsibility delegation, runtime behavior, and communication.
    • Ideal for achieving flexibility, reusability, and maintainability in complex systems.

    👁️ Observer Pattern¶

    ✅ Intent:¶

    • Defines a one-to-many dependency between objects so that when one object changes state, all its dependents are notified automatically.

    📦 Use Cases:¶

    • Event handling systems (e.g., GUI listeners)
    • Messaging systems (e.g., Kafka subscribers)
    • Real-time update scenarios (e.g., stock prices, dashboards)

    🧠 Key Concepts:¶

    • Subject: Maintains list of observers, notifies them of changes.
    • Observer: Defines the interface for receiving updates.

    🔧 UML Components:¶

    • Subject, ConcreteSubject, Observer, ConcreteObserver

    ✅ Pros:¶

    • Promotes loose coupling.
    • Allows dynamic subscriber management.

    ⚠️ Cons:¶

    • Complex dependency graph can lead to performance issues.

    🧠 Strategy Pattern¶

    ✅ Intent:¶

    • Enables selecting an algorithm at runtime by encapsulating them in separate strategy classes.

    📦 Use Cases:¶

    • Payment gateways (credit card, UPI, etc.)
    • Sorting algorithms
    • Compression strategies (ZIP, RAR, etc.)

    🧠 Key Concepts:¶

    • Context: Uses a strategy.
    • Strategy Interface: Defines the algorithm interface.
    • Concrete Strategies: Implement different versions of the algorithm.

    🔧 Code Sketch (Java):¶

    interface Strategy {
    int execute(int a, int b);
}
class AddStrategy implements Strategy {
    public int execute(int a, int b) { return a + b; }
}
class Context {
    private Strategy strategy;
    public Context(Strategy strategy) { this.strategy = strategy; }
    public int executeStrategy(int a, int b) {
        return strategy.execute(a, b);
    }
}

    ✅ Pros:¶

    • Reduces conditional logic.
    • Improves code flexibility and testing.

    ⚠️ Cons:¶

    • Overhead of multiple classes.

    🕹️ Command Pattern¶

    ✅ Intent:¶

    • Encapsulates a request as an object, thereby allowing users to parameterize, queue, and log requests.

    📦 Use Cases:¶

    • Undo/Redo operations
    • Task scheduling systems
    • GUI actions

    🧠 Key Concepts:¶

    • Command Interface: Declares execute().
    • ConcreteCommand: Implements the command.
    • Invoker: Triggers the command.
    • Receiver: Knows how to perform the operation.

    🔧 UML Flow:¶

    • Client → Invoker → Command → Receiver

    ✅ Pros:¶

    • Supports undo, logging, transaction systems.
    • Decouples sender and receiver.

    ⚠️ Cons:¶

    • Increased number of classes.

    🧬 Template Method Pattern¶

    ✅ Intent:¶

    • Define the skeleton of an algorithm in the superclass but let subclasses override specific steps.

    📦 Use Cases:¶

    • Frameworks that require extension (e.g., test runners)
    • Code generation or parsing

    🧠 Key Concepts:¶

    • Abstract Class: Implements the template method.
    • Concrete Class: Implements custom steps.

    🔧 Code Sketch (Java):¶

    abstract class DataProcessor {
    public final void process() {
        readData();
        processData();
        saveData();
    }
    abstract void readData();
    abstract void processData();
    void saveData() { System.out.println("Saving..."); }
}

    ✅ Pros:¶

    • Promotes code reuse and inversion of control.
    • Easy to enforce standard behavior.

    ⚠️ Cons:¶

    • Tight coupling with the base class.

    🧾 Summary Table¶

    Pattern Intent Key Participants Ideal Use Case
    Observer Notify dependents of state change Subject, Observer Real-time updates, Event systems
    Strategy Swap algorithms at runtime Context, Strategy Runtime algorithm flexibility
