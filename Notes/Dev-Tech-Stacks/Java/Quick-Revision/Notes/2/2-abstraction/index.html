    üî• Abstraction in Java¶

    ‚úÖ What is Abstraction?¶

    • Definition: Abstraction is the process of hiding implementation details and showing only the essential features of an object.
    • Purpose:
      • Simplifies complexity by exposing only relevant parts.
      • Achieves loose coupling between classes.
    • Real-World Example:
      • Car: You interact with the steering wheel and pedals but don‚Äôt need to know how the engine works internally.

    üî• Abstract Classes vs Interfaces¶

    ‚öôÔ∏è Feature üõ†Ô∏è Abstract Class üåê Interface
    Definition A class with abstract methods (incomplete methods) A collection of abstract methods (blueprint)
    Inheritance Supports single inheritance Supports multiple inheritance
    Method Implementation Can have both abstract and concrete methods Methods are abstract by default
    Default Access Can define protected and public members Only public members by default
    Static & Default Methods Not allowed Supported from Java 8+
    Use Case Used for common behavior with partial abstraction Used for full abstraction and contract enforcement

    üî• Abstract Classes in Java¶

    üìå Key Characteristics:¶

    • Declared using the abstract keyword.
    • Cannot be instantiated directly.
    • Can have abstract and non-abstract methods.
    • Can have constructors, variables, and methods.

    üõ†Ô∏è Syntax:¶

    abstract class Animal {
    abstract void makeSound(); // Abstract method
    void sleep() {              // Concrete method
        System.out.println("Sleeping...");
    }
}

class Dog extends Animal {
    void makeSound() {
        System.out.println("Bark");
    }
}

    ‚úÖ Use Case:¶

    • When you want to provide common functionality to multiple subclasses while enforcing them to implement specific methods.

    üåê Interfaces in Java¶

    üìå Key Characteristics:¶

    • Declared using the interface keyword.
    • All methods are abstract by default (before Java 8).
    • Supports multiple inheritance.
    • Cannot have constructors.
    • From Java 8 onwards, interfaces can have:
      • Default methods
      • Static methods

    üõ†Ô∏è Syntax:¶

    interface Vehicle {
    void start();          // Abstract method
    default void stop() {  // Default method (Java 8+)
        System.out.println("Vehicle stopped");
    }
    static void service() { // Static method (Java 8+)
        System.out.println("Vehicle service");
    }
}

class Car implements Vehicle {
    public void start() {
        System.out.println("Car started");
    }
}

    ‚úÖ Use Case:¶

    • When you want to define a contract that multiple classes must follow.
    • Useful for polymorphism and achieving loose coupling.

    üî• Default and Static Methods in Interfaces (Java 8)¶

    ‚úÖ Default Methods:¶

    • Introduced in Java 8 to allow adding new functionality without breaking existing classes.
    • Uses the default keyword.
    • Can have a method body.

    üõ†Ô∏è Syntax:¶

    interface Printer {
    default void print() {
        System.out.println("Printing...");
    }
}

    ‚úÖ Static Methods:¶

    • Introduced in Java 8.
    • Defined with the static keyword.
    • Can only be called using the interface name, not by implementing classes.

    üõ†Ô∏è Syntax:¶

    interface Logger {
    static void log(String message) {
        System.out.println("Log: " + message);
    }
}

// Usage
Logger.log("System error");

    üöÄ Key Differences Between Abstract Classes and Interfaces¶

    • Multiple Inheritance:
      • Abstract classes do not support multiple inheritance.
      • Interfaces support multiple inheritance.
    • Default Implementation:
      • Abstract classes can have constructors.
      • Interfaces cannot have constructors.
    • Performance:
      • Abstract classes are faster.
      • Interfaces introduce a bit of overhead due to dynamic method resolution.

    ‚úÖ When to Use What?¶

    • Use Abstract Class:
      • When you need common behavior across multiple classes.
      • When you want to partially implement functionality.
    • Use Interface:
      • When you want to enforce a contract.
      • When you need multiple inheritance.
      • For defining pure abstraction.

    üöÄ Best Practices:¶

    • Use interfaces for defining contracts and abstract classes for base behaviors.
    • Prefer interfaces for extensibility and flexibility.
    • Use default methods sparingly to avoid ambiguity in multiple inheritance.
    • Document static methods properly to avoid confusion.

    ‚úÖ Interview Tips:¶

    • Be prepared to explain the differences with examples.
    • Demonstrate knowledge of default and static methods in interfaces.
    • Understand when to use abstract classes vs interfaces in real-world scenarios.
    • Highlight multiple inheritance with interfaces.
