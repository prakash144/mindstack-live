    üöÄ SOLID Principles for Scalable & Maintainable System Design¶

    üéØ Purpose: Quick-reference, experience-aligned guide for senior engineers designing extensible OOP systems in Java ‚Äî LLD/HLD focused.

    1Ô∏è‚É£ Single Responsibility Principle (SRP)¶

    "A class should have only one reason to change."

    ‚úÖ Why It Matters¶

    • Promotes modularity and testability
    • Simplifies debugging and code reviews
    • Avoids tight coupling between unrelated responsibilities

    üí° Real-World Use Case¶

    PaymentService.java

    class PaymentProcessor {
    void process(Payment payment) {
        // Logic for processing payment
    }
}

class ReceiptGenerator {
    void generateReceipt(Payment payment) {
        // Logic for receipt generation
    }
}

    üîß Trade-Off: Extra classes, but better separation of concerns.

    2Ô∏è‚É£ Open/Closed Principle (OCP)¶

    "Software entities should be open for extension but closed for modification."

    ‚úÖ Why It Matters¶

    • Enables evolution without regressions
    • Supports plug-and-play architecture

    üí° Real-World Use Case¶

    interface PaymentStrategy {
    void pay(double amount);
}

class CreditCardPayment implements PaymentStrategy {
    public void pay(double amount) {
        // Credit card logic
    }
}

class UpiPayment implements PaymentStrategy {
    public void pay(double amount) {
        // UPI logic
    }
}

class PaymentService {
    private final PaymentStrategy strategy;

    PaymentService(PaymentStrategy strategy) {
        this.strategy = strategy;
    }

    void executePayment(double amount) {
        strategy.pay(amount);
    }
}

    üöÄ Outcome: Add new payment types without modifying PaymentService

    3Ô∏è‚É£ Liskov Substitution Principle (LSP)¶

    "Derived classes must be substitutable for their base classes."

    ‚úÖ Why It Matters¶

    • Ensures behavioral consistency in polymorphic usage
    • Enables reliable refactoring and subclassing

    üí• Violation Example:¶

    class Bird {
    void fly() {}
}

class Penguin extends Bird {
    void fly() {
        throw new UnsupportedOperationException();
    }
} // ‚ùå Violates LSP

    ‚úÖ Refactored Design¶

    interface Bird {}
interface FlyingBird extends Bird {
    void fly();
}

class Sparrow implements FlyingBird {
    public void fly() {}
}

class Penguin implements Bird {} // ‚úÖ LSP compliant

    üîÑ Use Case: Clean segregation prevents run-time failures.

    4Ô∏è‚É£ Interface Segregation Principle (ISP)¶

    "Clients should not be forced to depend on methods they do not use."

    ‚úÖ Why It Matters¶

    • Prevents bloating and leaky abstractions
    • Encourages focused contracts

    üí• Violation Example:¶

    interface Payment {
    void pay();
    void refund();
    void generateInvoice();
}

class CashPayment implements Payment {
    public void pay() {}
    public void refund() {}
    public void generateInvoice() {
        throw new UnsupportedOperationException();
    }
} // ‚ùå ISP Violation

    ‚úÖ Solution:¶

    interface Payable {
    void pay();
}
interface Refundable {
    void refund();
}
interface InvoiceGeneratable {
    void generateInvoice();
}

class CashPayment implements Payable, Refundable {}

    üß© Outcome: Granular interfaces fit specific needs.

    5Ô∏è‚É£ Dependency Inversion Principle (DIP)¶

    "High-level modules should not depend on low-level modules. Both should depend on abstractions."

    ‚úÖ Why It Matters¶

    • Enables loose coupling
    • Makes systems extensible and testable

    üí• Violation Example:¶

    class Keyboard {}
class Computer {
    private Keyboard keyboard = new Keyboard();
}

    ‚úÖ Refactored Design¶

    interface Keyboard {}

class WiredKeyboard implements Keyboard {}
class WirelessKeyboard implements Keyboard {}

class Computer {
    private final Keyboard keyboard;

    public Computer(Keyboard keyboard) {
        this.keyboard = keyboard;
    }
}

    üõ†Ô∏è Benefit: Easily switch or mock dependencies

    üß† Summary Table¶

    Principle Responsibility Impact on Design
    SRP One class = one responsibility üîÑ Easier updates, focused classes
    OCP Extend without modify ‚ûï Add features safely
    LSP Subtypes behave like base types ‚úÖ Reliable polymorphism
    ISP Fine-grained interfaces üîç Cleaner abstractions
    DIP Depend on abstractions üß± Decoupled, testable code

    üß∞ Bonus Tips for Implementation¶

    • Prefer composition over inheritance to maintain OCP and DIP.
    • Use Factory/Strategy patterns to align with OCP and DIP.
    • Avoid God objects ‚Äî SRP and ISP help here.
    • Integrate unit testing early ‚Äî SOLID naturally improves testability.
