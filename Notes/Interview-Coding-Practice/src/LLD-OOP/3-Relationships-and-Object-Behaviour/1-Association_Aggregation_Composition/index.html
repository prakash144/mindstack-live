    📘 Relationships Between Classes¶

    🧩 Purpose¶

    Understanding how classes relate helps model real-world systems and is crucial for crafting extensible, maintainable low-level and high-level designs.

    🔗 Types of Relationships¶

    1. Association (Has-A / Uses-A)¶

    • Definition: General binary relationship between two classes.
    • Key Design Use: Facilitates collaboration between objects without implying ownership.

    • Multiplicity Types:

      • One-to-One
      • One-to-Many
      • Many-to-Many
      • Implementation: Reference one class inside another.
    class Person {
    Passport passport; // One-to-one association
}

class Teacher {
    List<Student> students; // One-to-many association
}

class Student {
    List<Course> courses; // Many-to-Many (from Student side)
}

class Course {
    List<Student> students; // Many-to-Many (from Course side)
}

    2. Aggregation (Weak Has-A)¶

    • Definition: Specialized form of association with a "whole-part" relationship.
    • Ownership: Whole contains part but doesn't own it.
    • Lifecycle: Part can exist independently of whole.
    • Design Benefit: Loosely couples reusable objects.
    class Department {
    List<Employee> employees;
}

class Employee {
    String name;
}

    💡 Even if Department is deleted, Employee objects may persist elsewhere.

    3. Composition (Strong Has-A)¶

    • Definition: Stronger form of aggregation where part cannot exist without whole.
    • Ownership: Whole owns part.
    • Lifecycle: If whole is destroyed, part is destroyed.
    • Design Use: Strong containment when parts are not shareable.
    class House {
    private List<Room> rooms = new ArrayList<>();

    public House() {
        rooms.add(new Room("Living Room"));
        rooms.add(new Room("Kitchen"));
    }
}

class Room {
    private String name;
    Room(String name) {
        this.name = name;
    }
}

    🏠 Destroying a House means its Rooms also go away.

    🔄 Combined Real-World Scenario¶

    class Library {
    List<Book> books; // Aggregation: Books can exist elsewhere
}

class Book {
    List<Chapter> chapters; // Composition: Chapters tied to this book
}

    📊 Comparison Table¶

    Aspect Association Aggregation Composition
    Relationship General Weak Has-A Strong Has-A
    Ownership No Whole contains part Whole owns part
    Lifecycle Independent Part survives whole Part dies with whole
    Real-world Example Teacher ↔ Student Department ↔ Employee Car ↔ Engine

    🧠 Design Takeaways for Senior Engineers¶

    • Prefer association for loosely coupled modules (plugin, utility behavior).
    • Use aggregation when objects are reused or logically grouped but should remain decoupled.
    • Use composition when objects are part of a class's identity and not shared elsewhere.
    • Understand the trade-offs: tight vs. loose coupling, reusability vs. encapsulation.
    • Think in terms of ownership and lifecycle when choosing between these.

    Association, Aggregation, and Composition describe how classes are connected—whether they just know each other (Association), one uses another without owning it (Aggregation), or one fully owns and controls the other’s lifecycle (Composition).
