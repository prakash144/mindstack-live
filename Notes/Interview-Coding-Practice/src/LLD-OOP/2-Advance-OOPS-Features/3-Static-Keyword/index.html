    📌 Understanding the static Keyword in Java¶

    🎯 Purpose¶

    • Understand how static enables memory-efficient, class-level programming in Java
    • Apply static to utility design, shared configuration, and initialization patterns
    • Learn when and why to use static for clean LLD

    🧱 Static Members: Core Concepts¶

    🔹 Static Variables (Class Variables)¶

    • Declared with static keyword inside a class, outside methods
    • Shared across all instances of the class
    • Loaded once at class loading time
    • Useful for shared state, global counters, or configuration
    class Config {
    static int MAX_USERS = 100; // shared across all instances
}

    🔹 Static Methods¶

    • Bound to class, not instances
    • Can be called without creating an object
    • Can't access non-static members directly (no instance context)
    • Use case: utility, factory methods, stateless operations
    class MathUtils {
    static int square(int x) {
        return x * x;
    }
}

    🔹 Static Blocks¶

    • Executes once when the class is loaded
    • Used for initializing static variables with logic
    class AppConfig {
    static String ENV;
    static {
        ENV = System.getenv("ENV_MODE"); // read at class load time
    }
}

    🔹 Static Classes¶

    • Nested classes can be made static (no reference to outer class instance)
    • Used for grouping related helper logic
    class DBConnection {
    static class Builder {
        static DBConnection build() {
            return new DBConnection();
        }
    }
}

    🔄 Interaction: Static vs Non-Static¶

    Member Type Can Access Static Can Access Non-Static Requires Instance?
    Static Method ✅ Yes ❌ No ❌ No
    Non-Static Method ✅ Yes ✅ Yes ✅ Yes
    • Static members cannot access non-static members directly (no this context)
    • Non-static members can access both
    class Demo {
    static int counter = 0;
    int id;

    Demo() {
        id = ++counter; // valid: static used inside non-static
    }

    static void log() {
        // System.out.println(id); ❌ not allowed
    }
}

    ✅ Design Trade-Offs & Use Cases¶

    👍 Advantages¶

    • ✅ Memory efficiency: static variables initialized once
    • ✅ Utility methods: clean separation from instance behavior
    • ✅ Global config: single source of truth
    • ✅ Factory pattern: static builders for object creation

    🚫 Limitations¶

    • ❌ Tight coupling if overused
    • ❌ Difficult to mock/test static methods in unit tests
    • ❌ State management becomes complex in multithreaded environments

    🛠️ When to Use Static (Design-Driven View)¶

    Use Case Use Static? Reason
    Utility or helper method ✅ Yes No need for object state
    Shared configuration value ✅ Yes Common across instances
    Shared counter/id generator ✅ Yes One-time load & reuse
    Business logic tied to object ❌ No Needs instance-specific state
    Polymorphism/overriding ❌ No Static methods can't be overridden

    🧩 Real System Example¶

    Singleton + Static Utility¶

    class Logger {
    private static Logger instance;

    private Logger() {}

    public static Logger getInstance() {
        if (instance == null) instance = new Logger();
        return instance;
    }

    public void log(String msg) {
        System.out.println(msg);
    }
}
    • static method for singleton access
    • Avoids multiple logger instances (memory safe)

    📚 Recap¶

    • Use static to reduce memory footprint, simplify utility logic, and maintain shared state
    • Avoid it for anything that depends on object behavior, state, or needs test flexibility
    • Think in terms of ownership: if the data/logic belongs to the class (not instance), it should be static

    The static keyword defines class-level data or behavior, ensuring shared access across all instances and enabling utility-style design without object instantiation.
