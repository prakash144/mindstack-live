    üöÄ API Rate Limiter - System Design Notes¶

    üìö 1. What is a Rate Limiter?¶

    • Limits the number of events (requests) an entity (user, device, IP) can perform in a specific time window.
    • Example scenarios:
      • ‚úÖ 1 message per second per user.
      • ‚úÖ 3 failed credit card transactions per day.
      • ‚úÖ 20 accounts creation per day per IP.
    • üî• Purpose: Prevent abusive behavior and ensure service availability.

    üî• 2. Why do we need API Rate Limiting?¶

    • ‚úÖ Prevent DoS attacks: Block malicious request floods.
    • ‚úÖ Ensure fair usage: Prevent monopolizing resources.
    • ‚úÖ Reduce infrastructure costs: Avoid unnecessary scaling.
    • ‚úÖ Stop spam & online harassment: Block frequent abusive requests.
    • ‚úÖ Prevent bad design practices: Discourage inefficient API usage.
    • ‚úÖ Revenue model: Implement tier-based API access.
    • ‚úÖ Smooth traffic: Prevent sudden spikes to avoid service disruption.

    üéØ 3. Requirements and Goals¶

    ‚úÖ Functional Requirements:¶

    • Limit the number of requests per entity (e.g., 15 requests/second).
    • Rate limit enforced across a cluster of servers.
    • Return HTTP 429 - Too Many Requests on exceeding limit.

    ‚öôÔ∏è Non-Functional Requirements:¶

    • Highly available & fault-tolerant.
    • Minimal latency impact.
    • Scalable across multiple servers.

    ‚öôÔ∏è 4. How Rate Limiting Works?¶

    • Throttling: Controls API usage during a period.
    • Server Response: HTTP status 429 - Too many requests.

    üö¶ 5. Types of Throttling¶

    • üî• Hard Throttling: Strict limit, no exceeding beyond cap.
    • üî• Soft Throttling: Slightly flexible, e.g., 10% over limit.
    • üî• Elastic Throttling: Allows exceeding limits when resources are free.

    ‚öôÔ∏è 6. Rate Limiting Algorithms¶

    • Token Bucket
    • Leaky Bucket
    • Fixed Window Counter
    • Sliding Window Log

    üìä 1. Fixed Window Algorithm:¶

    • Time window: Fixed period (e.g., 0-60 seconds).
    • ‚úÖ Simple but can cause burst traffic issues at window boundaries.

    ‚è±Ô∏è 2. Rolling Window Algorithm:¶

    • Time window: Moves with each request (e.g., from T to T+1).
    • ‚úÖ More accurate rate limiting.
    • ‚úÖ Prevents boundary-related bursts.

    üõ†Ô∏è 7. High-Level Design for Rate Limiter¶

    • üåê Web Server ‚Üí Rate Limiter ‚Üí API Servers
    • ‚úÖ Rate Limiter decides whether to allow or reject requests.
    • ‚úÖ Ensures controlled and reliable request flow.

    ‚öôÔ∏è 8. Basic System Design & Algorithm¶

    üî• Hash Table Storage:¶

    • Key: UserID
    • Value: Struct with:
      • Count: Number of requests.
      • StartTime: Epoch time (request window start).
    • Algorithm Steps:
      1. If UserID not in hash table ‚Üí Insert with count 1 and current timestamp.
      2. If CurrentTime - StartTime >= 1 min ‚Üí Reset Count and StartTime.
      3. If Count < limit, allow request.
      4. If Count >= limit, reject request.

    ‚ö†Ô∏è Problems:¶

    • Atomicity issue: Concurrent updates can lead to race conditions.
    • Solution: Use Redis locks for atomic operations.
    • Memory Calculation:
      • UserID: 8 bytes.
      • Count: 2 bytes.
      • Time: 2 bytes.
      • Total: 12 bytes per record
      • For 1M users: 32MB memory usage.

    üî• 9. Sliding Window Algorithm¶

    • ‚úÖ More accurate request tracking.
    • ‚úÖ Uses Redis Sorted Set to store timestamps.
    • Algorithm Steps:
      1. Remove timestamps older than CurrentTime - 1 minute.
      2. Count timestamps ‚Üí Reject if above limit.
      3. Insert current timestamp ‚Üí Allow request.

    üî• Memory Calculation:¶

    • UserID: 8 bytes.
    • Epoch time: 4 bytes.
    • Overhead: 20 bytes for hash-table, 20 bytes for Sorted Set.
    • Memory per user: 12KB
    • For 1M users: 12GB memory usage.

    üî• 10. Sliding Window with Counters¶

    • ‚úÖ Combines Fixed & Sliding Windows.
    • ‚úÖ Uses multiple fixed windows (e.g., 1/60th of the main window).
    • ‚úÖ More memory efficient.

    üî• Algorithm Steps:¶

    • Store counters in Redis Hash with expiry.
    • On each request:
      • Sum counters for past hour.
      • Check against the threshold.
    • ‚úÖ Low memory footprint.
    • ‚úÖ Efficient distributed scaling.

    üî• Memory Calculation:¶

    • UserID: 8 bytes.
    • Epoch time: 4 bytes.
    • Counter: 2 bytes.
    • Memory per user: 1.6KB
    • For 1M users: 1.6GB memory usage.

    ‚öôÔ∏è 11. Data Sharding & Caching¶

    • ‚úÖ Sharding: Distribute users by UserID using Consistent Hashing.
    • ‚úÖ Caching:
      • Cache recent users for faster lookups.
      • Use Write-back cache for lower latency.
      • Apply LRU eviction policy.

    üî• 12. Rate Limiting by IP vs User¶

    üåê By IP:¶

    • ‚úÖ Simple but inaccurate.
    • ‚ö†Ô∏è Issues with shared IPs (e.g., cafes).
    • ‚ö†Ô∏è IPv6 attack vectors (large address space).

    üë§ By User:¶

    • ‚úÖ More accurate.
    • ‚úÖ Token-based authentication.
    • ‚ö†Ô∏è Vulnerable to DoS attacks against specific users.

    üî• Hybrid Approach:¶

    • ‚úÖ Combine both IP & User-based limiting.
    • ‚úÖ More accurate but higher memory usage.
