    📘 Quick Revision Notes: How PayPal Supported 1B Transactions/Day With Just 8 VMs¶

    🚀 A must-read for your Google SSE interview prep! Here's a quick, clean breakdown of the architecture and principles behind PayPal's billion-scale system.

    🏁 Initial Scaling Strategy¶

    • 🧑‍💻 Started as security software → Pivoted to online payments → Became PayPal.
    • 📈 Grew to 1M transactions/day in 2 years.
    • 🖥️ Horizontal Scaling with 1000+ VMs initially.

    😓 Challenges Faced:¶

    • 🕸️ Network Latency: More hops = Slower response time.
    • 💸 Infra Costs: Deployment, autoscaling, monitoring became expensive.
    • 🧃 Low Resource Utilization: CPU usage per server was suboptimal.

    🧠 The Actor Model to the Rescue¶

    Based on Akka framework (Java) — concurrency through message-passing and isolation.

    🔁 1. Concurrency & Threads¶

    • 🔹 Actors are super lightweight.
    • 🔹 A thread is assigned only when a message is being processed.
    • 🔹 Small thread pool → Manages millions of actors → Saves resources.

    🔐 2. Isolation & State Handling¶

    • 🔹 Actors do not share memory.
    • 🔹 Communicate via immutable messages.
    • 📦 Each actor has a mailbox (FIFO) for message queueing.
    • 🧠 Local state caching inside app server → Fewer DB/cache hits.
    • 🎯 Uses consistent hashing to route customers to same server.

    🕹️ 3. Asynchronous Execution¶

    • 🧵 Each actor processes only one message at a time.
    • 📬 Messages are processed asynchronously.
    • 🧩 Supports functional programming: Side-effect free, testable, scalable.

    🛠️ 4. Fault Tolerance¶

    • 👨‍👦 Actors can create & supervise other actors.
    • 🔄 On failure, supervisor restarts the child actor.
    • 🚨 Errors bubble up to the supervisor → Clean error handling.

    📌 Must-Know Concepts for Interviews¶

    🔥 1. Actor Model¶

    • 🧵 Concurrency: Efficient message-passing model avoids locking.
    • 🧠 Isolation: Private state; avoids race conditions.
    • 🔄 Fault Tolerance: Supervision model helps with graceful recovery.
    • 📬 Asynchronous: Fire-and-forget style leads to scalable systems.
    • 🛡️ Functional Programming: Stateless transformations = testable, scalable code.

    🔥 2. Efficient Resource Utilization¶

    • 💡 Threads are expensive → Actors are lightweight → Huge concurrency at low cost.
    • 💽 Application servers as stateful caches reduce latency & DB load.

    🔥 3. Consistent Hashing¶

    • 🎯 Ensures requests from a customer go to the same server.
    • ⚖️ Balances load and preserves cache locality.

    🏆 Key Takeaway¶

    The Actor Model + Functional Programming = 🚀 High concurrency, ⚡ low latency, 🔧 easy scalability, and 💪 fault-tolerant system.
